## Requiredness Semantics (Canonical)
1. `required_from_lv2`: field must exist in inbound LV2 payload.
2. `required_at_ingest`: ingest route rejects payload if field is missing/empty.
3. `required_in_db`: successful write requires field/invariant persisted.
4. `required_for_runtime`: export/player handoff requires field present.
5. `system_generated`: field is generated by CMS or DB and may be omitted by LV2.
6. `generator_owner`: one of `lv2`, `cms`, `db`, `unknown`.
7. Hard rule: if `required_for_runtime=true` or `required_in_db=true` and `system_generated=false`, then `required_from_lv2=true`.
8. No-assumption rule: missing evidence is `UNKNOWN` and is blocking until resolved.

## Generation Ownership Policy
1. DB-owned generated identity keys: `moduleId`, `lessonId`, `groupId`, `slideId`.
2. CMS-owned generated fields: `lastUpdatedAt`, `targetLanguage`; `orderIndex` is CMS-owned in `lesson_ends`.
3. `slug` is a canonical identity/meta key and is not URL-route-only metadata.
4. `slug` policy: derive from `title` on create only when missing; enforce uniqueness; do not auto-regenerate on update unless explicitly requested.
5. `sourceVersion` policy (current lane): default to `v1` when omitted.
6. `version` policy: CMS-owned revision counter (initialize on create; increment on accepted changed update/reingest).

## Canonical Naming Locks
1. Component tokens are plural snake_case (`modules`, `lessons`, `groups`, `slides`, `activity_slides`, `title_slides`, `lesson_ends`).
2. Top-level identity keys are singular camelCase + `Id` (`moduleId`, `lessonId`, `groupId`, `slideId`).
3. `slideId` is the only canonical update identity key for slide-family components.
4. `slideUuid` is forbidden.

## Hard Fact (HF-IDENTITY-001)
1. `slideId` is the only legal slide-family identity key in this repository.
2. `slideUuid` is forbidden in ingest payloads, export payloads, and runtime contracts.
3. No aliasing, fallback mapping, translation layer, or dual-key acceptance is allowed.

## Accepted Payload Rules
1. Payload root must be a JSON object with component-compatible categories.
2. Create mode must include the canonical parent key when parent linkage exists.
3. Update mode must include the canonical identity key.
4. Category objects may include only active fields for that component from `public.config_component_fields` (canonical read view).
5. Top-level identity keys must never appear inside category payload objects.
6. `activity_slides` must satisfy `propsJson` + `runtimeContractV1` envelope and ACT shape-lock rules.
7. `activity_slides` ingest uses whitelist validation derived from ACT shape-lock keys and canonical structured payload keys.
8. `runtimeContractV1.interaction` keys are strict snake_case: `activity_row_tool`, `command_row_controls`, `status`.
9. Supported ACT aliases/synonyms are accepted only when normalized to canonical payload keys/values before validation.
10. Imports are atomic per uploaded file (one failing row rejects the batch).
11. Non-activity component import requiredness (`modules`, `lessons`, `groups`, `slides`, `title_slides`, `lesson_ends`) is derived from `central/REQUIREDNESS_MATRIX.csv` `category_payload` rows and filtered to currently active config fields.

## Deterministic Rejection Envelope (JSON clients)
1. Import APIs return deterministic rejection payloads for non-HTML clients.
2. Canonical envelope v1 fields:
   - `code`, `component`, `operation`, `activityId`, `slideId`, `field`, `message`, `contractVersion`, `timestamp`
3. Non-activity import routes use generic import codes (`IMPORT_*`) from `lib/importGate.ts`.
4. `activity_slides` import route uses ACT-specific rejection codes (`ACTIVITY_*`) from `lib/activityIngestRejection.ts`.
5. HTML form clients continue to receive redirect-based error responses for `/import`.

## Rejected Payload Rules (Fail Closed)
1. Unknown component field for the target component.
2. Missing required key for create/update mode.
3. Legacy identity key usage (`slideUuid`) or unsupported alias payloads.
4. Top-level/category contamination (identity keys or blocked duplicates in category objects).
5. Missing required `activity_slides` envelope/shape for active ACT ids.
6. Unsupported or inactive `activityId` (`ACT-006`, `ACT-007`, `ACT-008`).
7. Runtime contract casing drift (for example `activityRowTool` / `commandRowControls` without canonical snake_case keys).

## ACT Normalization Notes
1. ACT-004 accepts `audioPrompt` input by canonicalizing to `propsJson.audio`; validation then checks playable audio via `audioId` or `audio.speech`.
2. ACT-004 intonation synonyms are normalized to canonical curve tokens before `correctCurveId` matching.
3. ACT-009/ACT-010/ACT-021 accept `correctAnswer` as label or 1-based index; ingest canonicalizes to label.
4. ACT-018 canonical gap shape is sentence-level objects with `gaps` rows that include `position` + `accepted_answers`. Ingest normalizes legacy aliases (`acceptedAnswers`, `correctGapIndex`, `index`, `acceptedAlternatives`) into canonical gap payload before validation.
5. ACT-023 accepts dialogue-turn audio as `audioFile` or `audio.speech`; ingest canonicalizes to canonical audio payload shape.
6. ACT-023 validation requires non-empty `avatarLine`, playable turn audio (`audioFile` or `audio.speech`), and at least one non-empty `correctResponses` value per turn.
